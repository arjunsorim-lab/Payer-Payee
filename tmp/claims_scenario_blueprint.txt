    Executive summary

         The scenario-map images are the front-end story: patient view, doctor view, provider view, money prediction,
          cost leakage, and savings phase.
         The Excel/CSV claims file is the data engine: patient, provider, CPT, ICD-10, units, charge, allowed, paid, patient
          responsibility, adjustment, status, denial reason, prior auth, and referral.
         The AI system should calculate money predictions with rules/ML, then use an LLM to explain the result in plain
          language and recommend provider actions.
         The provider view should be the primary decision screen because it connects claim quality, payment
          prediction, avoidable spend, and resubmission opportunities.




1. Product goal
Build a provider-focused claims intelligence app that connects Excel/CSV claims data to scenario-map dashboards and
predicts the money outcome of a claim or episode of care.

The app should answer:

        How much will the payer likely allow?
        How much will insurance likely pay?
        How much will become patient responsibility?
        How much will be adjusted/write-off?
        Will the claim be denied, forwarded, or processed?
        Where can the provider avoid repeated spend?
        Which phase creates the best savings opportunity?

7. Feature engineering formulas
 adjustment_rate = adjustment_amount / charge_amount
 allowed_rate = allowed_amount / charge_amount
 paid_rate = paid_amount / charge_amount
 patient_share_rate = patient_responsibility / allowed_amount
 has_prior_auth = prior_auth_number is not null and prior_auth_number != ""
 has_referral = referral_number is not null and referral_number != ""
 length_of_service_days = service_date_to - service_date_from + 1
 same_dx_repeat_count_30d = count claims for same member + diagnosis_group in prior 30 days
 days_since_last_same_dx = current_service_date - previous_same_diagnosis_service_date
 provider_denial_rate_90d = denied claims for provider in last 90 days / total claims for provider in last 90 days
 payer_allowed_rate_by_cpt = median(allowed_amount / charge_amount) grouped by payer_id + cpt_code
 expected_adjustment = charge_amount - predicted_allowed_amount

PREDICTION ENGINE
Start with deterministic rules and historical medians:
1. predicted_allowed_amount = charge_amount * median_allowed_rate for payer + CPT. Fallback to provider + CPT. Fallback to global median allowed_rate.
2. predicted_paid_amount = predicted_allowed_amount * median_paid_to_allowed_rate. Fallback global rate.
3. predicted_adjustment_amount = charge_amount - predicted_allowed_amount.
4. predicted_patient_responsibility = max(predicted_allowed_amount - predicted_paid_amount, 0), unless historical patient_share_rate exists.
5. denial_risk_score increases if:
 - CPT/ICD combination historically denied
 - missing prior auth for high-cost CPT or service category
 - missing referral where required
 - adjustment_rate is unusually high
 - same provider has high denial rate
 - denial reason patterns include CO-4, CO-97, CO-11
6. repeat_visit_risk_score increases if:
 - same member has same diagnosis group repeated within 30/60/90 days
 - symptoms/diagnosis not resolved in episode data
 - same CPT/diagnosis pattern repeats
7. estimated_avoidable_spend for episodes = estimated repeat cost that could be prevented by earlier triage, diagnosis, follow-up, or targeted treatment.

14. Accuracy limitations and data gaps
       The current claims file is enough for a prototype, but production predictions require thousands of historical
        claims across many patients, providers, payers, diagnosis groups, and procedure codes.
       Cavity/dental prediction requires dental-specific claim data. Medical 837 data alone is not enough for accurate
        dental cost prediction.


                          Claims Scenario Intelligence Blueprint | Provider-focused money prediction using claims data + AI/LLM
   Patient responsibility accuracy requires benefit design fields such as deductible, copay, coinsurance, out-of-
    pocket maximum, and secondary payer logic.
   Avoidable spend predictions require episode/outcome data such as resolved/not resolved, repeat visit windows,
    follow-up completion, and treatment response.
   LLM explanations should always state when data is missing and should not invent clinical or financial details.



15. Final product direction
The app should not be only a claims viewer. It should become a provider decision engine:

   Predict expected money before submission.
   Detect high-risk and high-cost claims.
   Explain why payment, adjustment, denial, or forwarding may happen.
   Show where avoidable spend can be reduced.
   Guide providers toward earlier diagnosis, better follow-up, cleaner claims, and lower total cost.
